# 开发复盘

## 一、Bug 分类统计

整个过程中出现了约 20+ 个 Bug，按根因分类：

### 🏗️ 1. 架构设计问题（占 40%）

| Bug | 根因 | 发生阶段 |
|-----|------|---------|
| 表单输入回车后菜名清空 | `renderEditForm()` 用 `innerHTML` 全量重绘，DOM 状态丢失 | 后期 |
| 粘贴封面图后字段清空 | 同上，`onCoverPaste` 调 `renderEditForm()` 重建 DOM | 后期 |
| 步骤 5+ 回车后步骤错乱 | 同上，多次重绘后索引偏移 | 后期 |
| 编辑已有菜谱显示新增菜谱的数据 | `editRecipe` 没清 DOM，`getFld` 读到残留值 | 后期 |
| Tab 栏挡住购物清单按钮 | `sub-page` 的 `z-index:400` 低于 tab 栏 `500` | 中期 |

**教训：** 用 `innerHTML` 做表单渲染是万恶之源。一开始就该用响应式方式（每个字段独立更新，或 Virtual DOM 思路），而不是每次重绘整个表单。后期所有 `renderEditForm` 相关的 Bug 都源于此。

### 🧩 2. 状态管理混乱（占 25%）

| Bug | 根因 |
|-----|------|
| 新增菜谱残留上一条数据 | `fabAction('edit')` 直接调 `page('edit')`，跳过了 `editRecipe(null)` 的初始化 |
| 删除食材后无法恢复 | `deleteCheckedIngredients` 用 `shop_excluded` 永久排除，无撤回机制 |
| 清空清单后重新添加食材仍未恢复 | `addToShopList` 没同时清理 `shop_excluded` 和更新 `shop_recipe_checked` |
| 剪切食材内容后回车仍出现旧值 | `onkeydown` 触发时剪切还没执行，`this.value` 是旧值 |

**教训：** 状态分散在多个全局变量（`editingRecipeId`, `editIngredientRows`, `editStepRows`, `editCoverDataUrl`），进入/退出路径不统一。应该有 klar 的生命周期：进入编辑 → 初始化状态 → 用户操作 → 保存 → 销毁状态。

### 🎨 3. CSS/布局问题（占 15%）

| Bug | 根因 |
|-----|------|
| FAB 按钮看不见 | `position:fixed;right:20px` 定位到浏览器视口右侧，不是居中 app 的右侧 |
| 购物清单底部按钮被盖 | `position:sticky` + `margin-top:auto` 在 flex 中行为不可靠 |
| 删除食材按钮无响应 | 按钮被 tab 栏（`z-index:500`）覆盖 |

**教训：** `max-width:430px;margin:0 auto` 的居中方式与 `position:fixed` 元素不兼容。fixed 元素需要用 `left:50%;transform:translateX(-50%)` 对齐。早期就应该统一定位方案。

### 🔍 4. 算法/解析问题（占 15%）

| Bug | 根因 |
|------|------|
| 文字解析拆成两道菜 | `parseTextRecipes` 看到短行就新建 recipe，看到食材又建第二个 |
| 导入不识别菜名 | 标题检测仅限 2-25 字，超过不匹配 |
| 食材不重复出现（牛肉牛肉牛肉） | `parseIngredient` 无量词时将原文设为 amount，下次编辑时 `name+amount` = 重复 |
| 调料未过滤 | `normalizeIngredient` 始终设 `category:'other'`，调料过滤条件从未触发 |

**教训：** 这些是典型的"第一版草稿问题"。算法逻辑写完后没有做边界测试。应该为 `parseTextRecipes` 和 `normalizeIngredient` 写单元测试。

### 🧹 5. 遗漏/缺失（占 5%）

- `clearShopAll` 函数从未定义（HTML 调用了但 JS 没有）
- 导入后文本框未清空（代码写到了但是 `finally` 前有异常就不会执行）

## 二、为什么会这么多 Bug？

### 2.1 开发方式：边聊边写，缺乏设计阶段

整个过程是「用户说需求 → 我直接写代码 → 用户试用 → 提 bug → 我修」。没有：
- 先画 UI 原型
- 先写核心算法测试用例
- 先设计数据流图

如果一开始就画出 `renderEditForm` 的调用链路，就会发现「每次重绘全量 DOM」的风险。

### 2.2 代码修改策略：打补丁式修改

修复 Bug 时倾向于最小改动：

```
Bug：编辑表单回车清空菜名
→ 修复：renderEditForm 加 getFld 从 DOM 读值
→ 新 Bug：新增菜谱读到上一条数据
→ 再修：editingRecipeId ? getFld : ''
→ 新 Bug：粘贴封面图又清空（editingRecipeId=null 时 getFld 被跳过）
→ 再修：改为进入 editRecipe(null) 时清 DOM
```

如果一开始就重构出清晰的表单状态管理（formState 对象 + 增量更新），这些连锁问题不会出现。

### 2.3 缺乏测试

整个项目没有任何自动化测试。每次修改靠手动点击验证，容易遗漏回归问题。

## 三、用户提的修改意见，有什么可以改进的？

### ✅ 做得好的地方

1. **需求表达清晰** —— 「待买清单中删除不影响后续重新添加」这类描述非常精准，容易定位代码
2. **优先级明确** —— 总是在说最影响使用的 Bug，没有被细节带偏
3. **勇于推翻** —— 删步骤/份量字段、改购物清单布局，这些是产品层面的正确决策

### 🔧 可以更好的地方

**1. 有些需求应该一开始就说清楚**

```
"仅保留菜名、子分类、食材、标签、备注"
```

如果能在一开始就确定这些字段，而不是在写了 7 个字段后再删 4 个，能省很多返工。

**建议：** 下次类似项目，第一步先敲定「添加菜谱的表单截图/草图」，确认每个字段后再开发。

**2. Bug 描述可以带上操作步骤**

```
Bug：步骤 5 回车后第四步内容没了
```

如果能附带复现步骤，定位效率会更高：
1. 新建菜谱
2. 添加 5 个步骤，每步填不同内容
3. 在第 5 步按回车
4. 第 4 步内容消失

**3. 复杂需求可以分批提**

```
需求1：删份量字段 + 需求2：删步骤字段 + 需求3：删来源链接 + 需求4：保留封面图
```

这四个需求看似独立但都改同一个表单。如果分两次提（先删再决定是否保留封面图），可以减少来回。

## 四、如果重来，我会怎么做

### 4.1 技术选型

```
当前                如果重来
innerHTML 全量重绘   → 每个字段独立 DOM 引用 + 函数更新
多个全局状态变量     → 单一 formState 对象
localStorage 直读     → 带版本号的 DAO 层
```

### 4.2 流程

1. **出 UI 草图** — 用 Figma/手绘确定每个页面的布局和字段
2. **定数据模型** — 先定 Recipe/Ingredient 接口，不改
3. **核心算法先测试** — `parseIngredient`、`generateShopList`、`parseSingleRecipe` 先写好测试用例
4. **再写 UI** — 数据层稳定后，UI 层只负责渲染和交互

### 4.3 心态

- 更早地说「这个改法会引发连锁问题，建议换个方案」
- 复杂功能（如图片裁剪、步骤自动编号）先给原型再细化
- 每次修改后跑一遍核心流程回归测试

## 五、总结

| 维度 | 评分（5分） | 说明 |
|------|:---------:|------|
| 需求理解 | ⭐⭐⭐⭐⭐ | 用户需求表达清晰，方向正确 |
| 响应速度 | ⭐⭐⭐⭐ | Bug 修复及时，但有连锁 Bug |
| 架构设计 | ⭐⭐⭐ | 表单渲染方案拖了后腿 |
| 测试覆盖 | ⭐⭐ | 手动测试为主，回归不充分 |
| 最终产品 | ⭐⭐⭐⭐ | 功能完整可用，值得花时间打磨 |

**最大的收获：** 先设计再编码的 ROI 在小项目中被严重低估 —— 看起来是「先写了再说更快」，实际上「先想清楚再写」在项目中期开始大幅领先。
